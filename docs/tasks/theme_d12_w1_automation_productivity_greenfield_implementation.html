Difficulty analysis

This task builds flowr, a greenfield CLI workflow orchestrator: read a YAML file describing a DAG of shell commands and run them in dependency order with bounded parallelism, retries, conditional steps, and dry-run, across 7 rounds. It is broad feature-plumbing work where finishing the whole feature set matters more than getting one numerical contract exactly right.

The widely-failed requirements are all later-round features: F12 ("unknown step key: retry"), F15 ("unknown step key: foreach"), F20 ("unknown flag: --dry-run"), plus F13, F16–F19, F25–F27 and the regression guard — each failed by 9 of the 10 models. The failure reasons are unusually concrete: the missing features are absent from the command-line surface entirely ("unknown step key" / "unknown flag"), not subtly wrong. Building the DAG core is achievable; layering retry/foreach/dry-run on top within the session is what the field cannot reach.

Performance gap

The usual hierarchy holds, but the bar is high. Claude-Opus-4.8 leads at 3/7 rounds (62% cases), clearing round 1 (81/81), round 2 (75/75) and round 5 (99/99) outright before the heavier later features stop it. GPT-5.5 builds the most breadth — 151/178 cases in round 3, 91/103 in round 4 — yet completes only round 1 (1/7, 69%): it is constantly a few cases short of a whole round. Qwen3.7-Max shows the same shape from below, accumulating cases (143/178 in round 3, 88/111 in round 7) without ever finishing a round (0/7, 67%).

The rest finish zero rounds with partial orchestrators: GLM-5.1 and Qwen3.6-Plus clear most of round 1 (74/81, 64/81) then stall at 34–36% overall, and MiniMax-M3, the DeepSeek pair and MiniMax-M2.7 sit at 21–28%, implementing a fraction of the command surface. At the bottom, Kimi-K2.6 never produces a working flowr binary at all — every round fails the build gate ("flowr binary missing from PATH", 0%). The round score and the case score agree here: this is a hard, honest from-scratch build where only the strongest model lands more than one round.
