Difficulty analysis

This task builds flowr, a greenfield CLI workflow orchestrator: read a YAML file describing a DAG of shell commands and run them in dependency order with bounded parallelism, retries, conditional steps, and dry-run, across 7 rounds. It is broad feature-plumbing work where finishing the whole feature set matters more than getting one numerical contract exactly right.

The widely-failed requirements are the later-round features. F17 and F26 (9/10 models each), F15 (8/10), and F7, F13 and the regression guard (7 each). The failure reasons make the cause unusually concrete: the strong models simply did not implement features — Opus returns "rc: 4 (err: unknown flag: --dry-run)" on F20 and "unknown step key: foreach" on F15, i.e. the command-line surface is missing entirely, not subtly wrong.

Performance gap

This task inverts the usual hierarchy. The leaders are the fast/cheap tier: DeepSeek-V4-Flash and MiniMax-M2.7 both reach 5/7 rounds at 100% case pass, losing only one or two cases each (F17, F26). GLM-5.1 follows at 4/7 (85%). These models implemented the full feature set and ran every step cleanly.

The flagships collapse. GPT-5.5 manages only 1/7 at 59%, and Opus-4.8 scores 0/7 at just 47%, both bleeding cases on F7 (35–36), F20 (31) and F13 — features they never wired up (the "unknown flag" / "unknown step key" errors), plus the regression guard (28). MiniMax-M3 is worst at 19%, with F28/F27 and regression all broken, and the Qwen pair and DeepSeek-V4-Pro sit at 27–38%. The lesson here is not partial-progress hidden by binary rewards — it is the opposite: the small models genuinely built more of the orchestrator, and the round scores reflect it honestly.
