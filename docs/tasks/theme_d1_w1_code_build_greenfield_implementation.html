Difficulty analysis

This is a greenfield build of jobforge, a Go CLI that runs a set of named jobs as a DAG: validate the config, compute a hash, schedule jobs onto slots, report run results, and answer graph queries (critical path, ancestors, impact). 9 rounds, growing test surface. Because it starts from nothing, a model must invent the whole command surface and output contracts at once, and the verifier is all-or-nothing per round.

The defining feature is a large common-failure core hit by all 10 models: schedule-assign, schedule-output, critical-path, critical-output, critical-tiebreak and ancestors-subcmd (each 10/10). The downstream graph queries impact-subcmd and ancestors-subcmd top nearly every model's fail list (13 and 11 cases). Reasons show two failure shapes: weaker models simply stub the command ("err=jobforge: not implemented" on err-precedence), while stronger ones get the wiring but the semantics wrong — run-signal returning "ec=129 st=failed" where exit_code 143 and status failed were expected, or err-precedence reporting the wrong validation error first.

Performance gap

GPT-5.5 leads at 5/9 rounds (69% case pass), followed by DeepSeek-V4-Pro 4/9 (60%) and Opus-4.8 only 2/9 (51%) — an inversion of the usual frontier ranking, suggesting this output-contract-heavy task rewards GPT-5.5's literalism. Qwen3.7-Max and DeepSeek-V4-Flash each manage 1/9 (50% / 41%).

The bottom tier — Kimi-K2.6, MiniMax-M3, Qwen3.6-Plus, GLM-5.1, MiniMax-M2.7 — all score 0/9 rounds, but case pass exposes how differently they fail: Kimi and MiniMax-M3 still pass 56% / 53% of cases (near the leaders, but never a clean round), while GLM-5.1 collapses to 6% and MiniMax-M2.7 to 31%. Crucially, every model fails the same scheduling/critical-path/ancestors core, so the separation is entirely in the surrounding breadth — how many of the other subcommands a model wires correctly — not in cracking the hard core, which none of them do.
