Difficulty analysis

This is a greenfield build of jobforge, a Go CLI that runs a set of named jobs as a DAG: validate the config, compute a hash, schedule jobs onto slots, report run results, and answer graph queries (critical path, ancestors, impact). 9 rounds, growing test surface. Because it starts from nothing, a model must invent the whole command surface and output contracts at once, and the verifier is all-or-nothing per round.

Across 12 models the task exposes two distinct failure shapes: weaker models simply stub or misroute subcommands, while stronger ones get the wiring right but slip on output-contract semantics — scheduling slot assignments, critical-path tiebreaking, or the exact signal-exit encoding expected by the verifier. The result is a wide spread from perfect completion down to near-zero, with no single requirement acting as a universal wall.

Performance gap

Opus-4.8-xhigh and DeepSeek-V4-Pro both clear all 9/9 rounds at 100% case pass, followed closely by MiniMax-M3 and Qwen3.7-Max at 8/9 (99% and 100% case pass respectively) and Qwen3.6-Plus and GLM-5.2 at 7/9. GPT-5.5 and DeepSeek-V4-Flash each land at 5/9, with DeepSeek-V4-Flash notably higher on case pass (98% vs 69%), indicating GPT-5.5 fails rounds more cleanly while DeepSeek-V4-Flash passes more cases within failed rounds. MiniMax-M2.7 reaches 4/9 at 91%.

The bottom tier is sharply divided: GLM-5.1 stalls at 0/9 rounds but retains 64% case pass, suggesting partial subcommand wiring without ever completing a full round cleanly, while Kimi-K2.7-Code collapses to 0/9 and only 16% case pass — the weakest result across the field and a clear outlier even among non-completing models.
