Difficulty analysis

This task builds mlpipe, a Go CLI for orchestrating ML experiment pipelines: plan a DAG, run stages, sweep hyperparameters, compare and report results, and export to JSON/CSV — 7 rounds, 68 → 427 cumulative test cases. Because the round reward is all-or-nothing across every active requirement, a single missing feature zeros the round even when nearly everything else passes.

The load-bearing failure is artifacts, missed by 9 of 10 models: the verifier expects each run to record produced files and the implementations leave the list empty ("artifacts: list has entries (got )"). The other shared misses are export-json (7 models — e.g. "export: json num_samples_norm is null (got 0.5)"), sweep-validate (6 — "sweep: invalid metric error message"), run-params, report-only and export-sort. These are output-plumbing and validation details layered on top of an otherwise working orchestrator.

Performance gap

The strong tier runs the whole pipeline and loses almost nothing else. Opus-4.8 and GPT-5.5 sit at 97% case pass yet score 0/7 — Opus's only listed miss is artifacts (28 cases), GPT-5.5 adds sweep-validate (10). MiniMax-M3 (90%) trails slightly with run-metrics, compare-weighted and export-json failures on top of artifacts. Kimi-K2.6 actually clears 2/7 rounds at 95% — the only model to pass any — but hits 2 build✗ rounds where its own code failed to compile.

The weak tier fails a broader, more basic layer. Qwen3.7-Max (37%) and MiniMax-M2.7 (40%) lose plan-order, plan-grid and run-diamond — the DAG planning and diamond-dependency execution itself — while GLM-5.1 collapses to 12% with 3 build✗ rounds and broad report/plan failures. DeepSeek-V4-Flash (29%) and the two 62% Qwen/DeepSeek-Pro models round out the spread. So the gap is layered: strong models implement the orchestrator and lose only artifact/export edge contracts, while weak models never get the planning and execution core working.
