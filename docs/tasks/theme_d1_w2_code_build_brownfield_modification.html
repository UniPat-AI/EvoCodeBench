Difficulty analysis

This is a hard brownfield task: minicc is an existing bytecode compiler and stack-based VM for a small language called Mini, and the agent must evolve it — adding language features like for-in/for-of loops, closures, exceptions and hashmaps to a codebase whose lexer, parser, codegen and VM already have fixed contracts. 8 rounds. Brownfield raises the bar: new features must thread through every layer of the existing pipeline without breaking what already passes.

The commonly-failed requirements are real language semantics, missed by all 12 models: capture-scope (12/12) and existing-programs (12/12). At 11 of 12 models: nested-call-frames, closures, closures-higher-order and array-push. The failure reasons are crisp wrong answers from the VM: capture-scope yields "actual: 999 1" where "999 42" was expected (closures not capturing the correct scope), existing-programs drops the last token ("actual: 1 50" vs "1 50 999"), and nested-call-frames loses the leading element from the expected sequence — regressions showing models break the pre-existing test suite while adding features.

Performance gap

This task produces the steepest single-model lead in the set. Opus-4.8 scores 6/8 rounds at 99% case pass, clearing nearly every requirement across six consecutive rounds. Only two other models clear even one round: GPT-5.5 at 1/8 (57%) and DeepSeek-V4-Pro at 1/8 (49%).

The remaining 10 models all land at 0/8 rounds, with case rates ranging from Qwen3.6-Plus at 9% up to MiniMax-M3 at 47% — wide variation but no clean rounds anywhere. The separation is almost entirely Opus vs. the field: only Opus consistently handles closure semantics, capture-scope correctness and nested call frames across rounds; everyone else gets the happy path but breaks on the stateful, scoping-sensitive constructs that define the later rounds.
