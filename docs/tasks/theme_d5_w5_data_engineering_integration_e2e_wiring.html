Difficulty analysis

This task stands up flowctl, a CLI that reads a YAML description of a data pipeline and actually runs it end-to-end: chained stages (group, pivot/unpivot, window, bucket, rank, join) over CSV/NDJSON, 9 rounds growing from a handful to dozens of cumulative cases. The work is integration wiring — each stage must compose with the next, and the config parser must accept or reject every stage's options exactly.

Across 12 models the hardest requirements are stage-level correctness: summary-dropped-late and window-slide each block 9 models, while window-emit-schema, sort-missing-key, group-aggregates, and group-by each block 7. The failure reasons show that most models either never implement the grouping or windowing stages at all — the parser returns "unknown stage type: group" — or they mishandle late-data summary fields and sliding-window emission.

Performance gap

The separation is sharp. Opus-4.8-xhigh (6/9 rounds, 88% cases) is the only model to wire most of the pipeline together. GPT-5.5 and Qwen3.7-Max reach 4/9 (56%) and 2/9 (58%) respectively, and DeepSeek-V4-Flash manages 1/9 (42%) — these four are the only models that pass any rounds. DeepSeek-V4-Pro and GPT-5.5 share the same 56% case rate yet DeepSeek-V4-Pro passes 0 rounds, illustrating how partial-stage output inflates case counts without completing any full round.

The remaining 8 models — Kimi-K2.7-Code (42%), Qwen3.6-Plus (43%), GLM-5.2 (39%), Kimi-K2.6 (23%), GLM-5.1 (28%), MiniMax-M2.7 (15%), MiniMax-M3 (13%), and DeepSeek-V4-Pro (56%) — all score 0/9 rounds and collectively stall on the same block: summary-dropped-late, window-slide, group-aggregates, and group-by. The task cleanly divides models that can build a composable pipeline from those that implement only a partial parser.
