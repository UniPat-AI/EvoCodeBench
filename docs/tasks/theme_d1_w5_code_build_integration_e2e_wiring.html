Difficulty analysis

This task wires up buildctl, a Go CLI that orchestrates multi-target build pipelines: dependency resolution, parallel execution, caching, hooks, variable substitution and config imports. 8 rounds across 12 models. As an integration / e2e-wiring task, the difficulty is making independently-correct pieces work together — affected-set computation feeding the planner, hooks chaining in order, variables resolving with the right precedence across imported files.

The hardest commonly-failed area is profiling and matrix expansion: profile-critical-path is missed by 10 of 12 models ("expected='a,b,c', got=''"), while matrix-include, matrix-exclude, matrix-expansion, retry.matrix and retry.imports are each missed by 9 of 12 models — every late-round feature lands as a cluster failure, meaning the wiring between matrix state and retry/import logic is almost universally broken.

Performance gap

GPT-5.5 stands alone at 6/8 rounds and 96% case coverage — the only model to push through matrix expansion and retry rounds. The next tier, Opus-4.8-xhigh and GLM-5.2, each pass 2/8 rounds (96% and 82% cases respectively), showing solid core construction but stalling at the same matrix/profile wall the field hits. profile-critical-path, matrix-include, matrix-exclude, matrix-expansion, retry.matrix and retry.imports are the load-bearing failures keeping every model below GPT-5.5.

High case coverage masks shallow completeness throughout: Kimi-K2.6 (99% cases, 1/8 rounds), Qwen3.6-Plus (97% cases, 1/8 rounds) and Opus-4.8-xhigh (96% cases, 2/8 rounds) all build a nearly-correct core but cannot integrate the late features into a passing round. At the bottom, MiniMax-M2.7 scores just 10% on cases and 0/8 rounds — a structural failure, not a feature gap.
