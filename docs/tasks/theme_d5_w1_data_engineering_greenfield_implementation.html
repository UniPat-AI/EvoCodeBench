Difficulty analysis

This is a long (13-round) greenfield data-engineering build: eventq, a CLI that turns raw event logs into clean, currency-normalized summaries — parsing, deduplication, sessionization, currency conversion via FX rates, sharded reporting, and reproducible run provenance. The 13-round chain accumulates a deep contract surface, and each round's all-or-nothing verifier re-checks everything built so far.

Across 12 models the hardest commonly-failed requirements cluster in the reporting and provenance layers: sharding-report, sharding-histogram, and time-range-observed each fail 11 of 12 models, while summary-schema-out, schema-provenance, and schema-digest fail 10 of 12. The failure modes split between silent wrong values — sharding-report and time-range-observed produce empty output — and hard exits: schema-digest exits 2 with "ingest requires --input-dir, --rates, --output, --rejects, --session-gap, --watermark-lag", and summary-schema-out similarly exits 2 rather than emitting the expected sidecar.

Performance gap

This is a one-model task. Opus-4.8 reaches 10/13 rounds at 95% case pass, losing only the latest sharding and time-range cases. The drop-off is immediate and severe: GPT-5.5 is second at 3/13 and 49% case pass, followed by Qwen3.7-Max at 2/13 and 43%. Three models — Kimi-K2.7-Code (23%), DeepSeek-V4-Pro (15%), and GLM-5.1 (22%) — each pass exactly 1/13 rounds.

The remaining six models all score 0/13 rounds, with case rates confirming genuine failure rather than hidden progress: Kimi-K2.6 32%, GLM-5.2 9%, Qwen3.6-Plus 14%, MiniMax-M2.7 5%, and DeepSeek-V4-Flash and MiniMax-M3 both at just 3%. Everyone outside Opus collapses on the same sharding/provenance/schema reporting core, so the gap is less about which subtle requirement a model misses and more that only Opus sustains a correct, reproducible pipeline across all 13 rounds.
