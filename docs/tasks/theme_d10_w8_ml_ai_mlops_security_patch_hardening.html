Difficulty analysis

This is a security-hardening task: build a complete ML model-serving platform as one Go binary — an HTTP gateway with authentication, role-based authorization, audit logging, input validation and request-size limits guarding model create/upload/deploy/inference — 8 rounds across 12 models. Unlike pure data-transformation tasks, much of the difficulty is getting the auth and validation middleware exactly right, because every protected endpoint depends on it.

The single most commonly-failed requirement in the new evaluation is model-create (7 of 12 models), where implementations accept a duplicate model registration rather than returning 400/409/422 — a too-permissive conflict check that lets bad state propagate silently through the rest of the suite.

Performance gap

The top cluster in this 12-model evaluation is anchored by Opus-4.8 at 8/8 rounds and 100% cases, with MiniMax-M3 (7/8, 100%), GLM-5.1 (7/8, 99%), MiniMax-M2.7 (7/8, 99%), and DeepSeek-V4-Flash (7/8, 96%) close behind — all five get the auth and validation middleware working end-to-end and lose only the narrowest edge in model-create duplicate enforcement. Kimi-K2.6 holds at 5/8 (97% cases), clearing the auth gates but dropping three rounds to the same conflict-check gap.

Six models pass zero rounds. GPT-5.5 (0/8, 68%) and DeepSeek-V4-Pro (0/8, 66%) retain partial case coverage, and Kimi-K2.7-Code reaches 82% cases without a single full-round pass, but Qwen3.6-Plus (0/8, 0%), GLM-5.2 (0/8, 0%), and Qwen3.7-Max (0/8, 55%) fall at or below the zero-cases threshold. The binary round reward is unforgiving here: hardening middleware — particularly model-create conflict handling — is the single gate separating the clusters.
