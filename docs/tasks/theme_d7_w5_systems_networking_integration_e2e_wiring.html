Difficulty analysis

This task builds a Go service-mesh health router from scratch: three integrated components (health monitor, traffic router, rate limiter) wired over a Unix-socket IPC, accumulating across 8 increasingly demanding rounds. The 30-minute per-round agent time budget acts as a hard capability wall — weaker models exhaust the budget mid-chain and the run aborts before later rounds can execute, while stronger models sustain enough implementation quality to survive all 8. Because each round's verifier checks every still-active requirement and the round reward is all-or-nothing, even models that build most of the system score 0 rounds if a single sub-feature misfires. Across 12 models, only GPT-5.5 clears one round, and none clear more.

The load-bearing failing requirements — each blocked for the majority of models — are: rate-limit-queue, rate-limit-burst, rate-limit, retry, rate-limit-stats, and router-proxy. These are stateful, timing-sensitive behaviours in the rate-limiter and retry subsystems — exactly what tends to break when an agent implements the happy path but not the full state transitions under concurrent load.

Performance gap

A clear capability wall separates two clusters. GPT-5.5 (84% cases, 1 round) and Opus-4.8-xhigh (86% cases, 0 rounds) build virtually the entire system and only fail on specific timing-sensitive requirements — their high case scores show the implementation is mostly correct. DeepSeek-V4-Pro reaches 54% before the chain stalls, while GLM-5.1, Kimi-K2.6, and Qwen3.6-Plus cluster in the 33–39% range. The remaining six models — MiniMax-M3, Qwen3.7-Max, MiniMax-M2.7, GLM-5.2, DeepSeek-V4-Flash, and Kimi-K2.7-Code — score between 0% and 20%, indicating the agent exhausted the per-round time budget before meaningful progress on the rate-limiting and retry machinery.

The universal blocker across tiers is the rate-limiting subsystem: rate-limit-queue, rate-limit-burst, and rate-limit are failed by 9 of 12 models, and retry and rate-limit-stats follow the same pattern. Even the top two models fail these requirements at the case level, confirming that correct concurrent rate-limiter implementation — not round completion — is what separates scores at the high end of this 86-point spread.

Is the task fair? (top-model check)

Opus-4.8 runs all 8 rounds without hitting the time budget (unlike weaker models that abort mid-chain) but scores 0 rounds, failing at Round 1 — "Service Mesh Health Router" — with 40/45 cases passing. The five failing cases expose three requirements: health-monitoring returns a blank healthy-count string where 3 api-gateway endpoints are expected, circuit-breaker emits an empty state field instead of half_open, and router-proxy exits cleanly but returns empty output rather than echoing the payload. These are self-consistent, well-specified checks that the oracle solution passes, so the failures reflect genuine gaps in Opus's wiring between the health-monitor IPC layer and the verifier's output format — a capability ceiling on this benchmark's hardest task, not a test defect.
