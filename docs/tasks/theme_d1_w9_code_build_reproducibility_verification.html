Difficulty analysis

This task builds buildvault, a deterministic build-artifact verification tool in Go (a skeleton with go.mod is provided): it must produce reproducible content hashes, lock dependencies, verify rebuilds match, run selective builds over a dependency closure, and maintain a tamper-evident audit chain. 7 rounds. Reproducibility is the theme, so the bar is exactness — byte-identical hashes, stable lockfiles, chained hashes that validate.

The single dominant obstacle is hooks (failed by 9 of 10 models), along with hooks-changed, hooks-fingerprint and hooks-lockfile — the hook system that runs commands around builds and folds their state into the fingerprint. The reasons are blunt: hooks reports "marker file not found" (the hook never executed), verify-rebuild emits empty "cached_hash= fresh_hash=" fields, audit-chain returns "got 0 chars" where a 64-char chain hash was required, lock-cmd exits 1, and constraints exits 0 where exit 1 was expected. selective-build and check-change-reasons round out the shared misses.

Performance gap

Opus-4.8 solves it outright: 7/7 rounds, 100% case pass — the only model to produce a clean run. Kimi-K2.6 is a strong second at 5/7 (93%), losing mainly to hooks (30 cases) and the export commands. GPT-5.5 and Qwen3.7-Max manage 2/7 (72% / 83%).

The remaining six models all score 1/7 or 0/7 rounds, but case pass shows real stratification rather than uniform failure: Qwen3.6-Plus and GLM-5.1 still clear 84% / 82% of cases (one passing round, then blocked by hooks), while MiniMax-M3, DeepSeek-V4-Pro and DeepSeek-V4-Flash sit near 46-49% — the lock-cmd/verify-rebuild/hooks stack failing wholesale. The discriminator is narrow and specific: nearly every non-Opus model implements the core verifier but cannot get the hook subsystem to execute and fold into the fingerprint, and that one missing piece caps the round score regardless of how much else works.
