Difficulty analysis

This task builds buildvault, a deterministic build-artifact verification tool in Go (a skeleton with go.mod is provided): it must produce reproducible content hashes, lock dependencies, verify rebuilds match, run selective builds over a dependency closure, and maintain a tamper-evident audit chain. 7 rounds across 12 models. Reproducibility is the theme, so the bar is exactness — byte-identical hashes, stable lockfiles, chained hashes that validate.

The dominant obstacles span several subsystems: export-cmd exits 1 (11 models), while audit-chain returns empty output where a 64-char hash was required, and selective-build, status-cmd, hooks, and export-chain each fail 10 of 12 models — together forming a wall that prevents most models from progressing past the first few rounds regardless of how much core verifier logic is correct.

Performance gap

Opus-4.8 solves it outright: 7/7 rounds, 100% case pass — the only model to produce a clean run. GPT-5.5 is a clear second at 3/7 rounds and 76% cases. Qwen3.6-Plus (2/7, 91%) and DeepSeek-V4-Flash (2/7, 71%) manage two rounds each, showing that correct early-round implementations do not carry through once export-cmd and audit-chain are introduced.

The remaining eight models score 1/7 or 0/7 rounds, yet case pass reveals real stratification: MiniMax-M3 reaches 93% and Qwen3.6-Plus 91% on case coverage within their passing round, while GLM-5.2 and MiniMax-M2.7 collapse to 14% and 36% respectively — the selective-build, hooks, and export-chain stack failing wholesale and pulling case scores down with them. The discriminator is specific: nearly every non-Opus model implements the core hash-and-lock verifier but cannot produce a working artifact export or audit chain, and those two missing pieces gate every subsequent round.
