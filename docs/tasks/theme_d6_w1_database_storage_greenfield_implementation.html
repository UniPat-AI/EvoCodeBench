Difficulty analysis

This is a from-scratch greenfield build of lsm-kvstore: a high-performance embedded key-value engine in Go using an LSM-Tree — write-ahead log, memtable, on-disk SSTables, compaction/merge, MVCC snapshots, transactions, TTL expiry and prefix compression, 8 rounds, 45-plus cumulative cases. This is the deepest systems task in the set, and no model passes a single round: the round table is all zeros.

Two failure modes dominate. First, the binary itself frequently fails to compile — Kimi-K2.6, MiniMax-M3, DeepSeek-V4-Flash (2 build✗ each), Qwen3.6-Plus and DeepSeek-V4-Pro (5 build✗), and MiniMax-M2.7 never compiles a single round (8 build✗, 0% cases). The reference oracle builds fine, so these are the models' own compile errors, not a task defect. Second, even compiling models share precise behavioural bugs: the on-disk formats in wal-batch-format / wal-binary-format, the verify-command that returns "ERROR: unknown command" because it was never implemented, MVCC merge ordering (merge-persistence emitting "a,a,b" instead of "a,b"), and a striking TTL bug where the CLI swallows the flag as data — ttl-batch-put stores the literal "bv1 --ttl 1" instead of expiring the key.

Performance gap

With every model at 0/8, the only meaningful axis is case pass, and it spreads from 98% down to 0%. Opus-4.8 leads at 98% cases (ran all 8 rounds, no build failures), losing only narrow items — verify-command (42 cases), tx-delete-range-conflict and checksum-compaction. GPT-5.5 (89%) and Qwen3.7-Max (83%) follow, their misses concentrated in the WAL/SSTable binary formats and the TTL CLI-parsing bug.

The weak tier is defined less by logic and more by whether the code builds: GLM-5.1 (75%) ran all rounds but failed the WAL formats; Kimi-K2.6 (85%), DeepSeek-V4-Flash (63%), MiniMax-M3 (62%), DeepSeek-V4-Pro (61%) and Qwen3.6-Plus (76%) all carry 2–5 build✗ rounds that drag their effective coverage down; and MiniMax-M2.7 sits at 0% — eight rounds, eight failed builds, never a working binary. The takeaway: a from-scratch LSM engine is hard enough that the differentiator is simply keeping the program compiling and getting the byte-level on-disk formats right, and not one model managed a fully clean round.
