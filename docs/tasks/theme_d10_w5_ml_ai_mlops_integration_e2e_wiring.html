Difficulty analysis

This task wires up mlpipe, a Go end-to-end ML pipeline orchestrator: plan and run DAG stages, train with multiple algorithms, cross-validate, build ensembles, and manage a model registry with promotion and lineage — 7 rounds, 48 → 130 cumulative cases. Because the round reward is all-or-nothing across every active requirement, a single persistent miss zeros the round — and here it zeros all 10 models.

The universal blocker is registry-promote, the most common failure by far (10 models): promoting a model to a stage doesn't take effect and the verifier sees the wrong count ("assertion failed (actual=1)"). Right behind it is ensemble-stage (9 models, same assertion signature), then promote-force (6), cv-validation (5), skip-continue-cascade (5) and algo-gradient-boost (5). The promotion and ensemble-staging logic is the shared wall.

Performance gap

The round table is a complete wall of zeros, so case pass is the only meaningful axis. Opus-4.8, GPT-5.5, Kimi-K2.6 and MiniMax-M3 all sit at 97–98% case pass — they build essentially the entire orchestrator and lose only the promotion/ensemble edges: Opus and GPT-5.5 fail just registry-promote (7 cases) and ensemble-stage (3), Kimi adds cv-validation (12), MiniMax-M3 adds algo-gradient-boost (7). DeepSeek-V4-Pro (94%) and Qwen3.7-Max (93%) trail just behind with wider registry-promote and lineage misses.

The weak tier fails the pipeline core itself. Qwen3.6-Plus (69%) and DeepSeek-V4-Flash (48%) lose cv-execution (30 cases) and stage-evaluate — cross-validation and stage scoring never run correctly. GLM-5.1 (26%, 2 build✗) and MiniMax-M2.7 (31%, 3 build✗) are worst, with their own code failing to compile across rounds and broad plan-command/transform-ops failures. So the gap is layered: the strong four are within one or two requirements of a clean run that the binary reward erases entirely, while the weak four never get cross-validation and DAG execution working.
