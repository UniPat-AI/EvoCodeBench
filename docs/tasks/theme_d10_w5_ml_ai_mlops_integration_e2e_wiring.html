Difficulty analysis

This task wires up mlpipe, a Go end-to-end ML pipeline orchestrator: plan and run DAG stages, train with multiple algorithms, cross-validate, build ensembles, and manage a model registry with promotion and lineage — 7 rounds, 48 → 130 cumulative cases. Because the round reward is all-or-nothing across every active requirement, a single persistent miss zeros the round — and here it zeros all 12 models.

The universal blockers are registry-promote and ensemble-stage, each failing 11 of 12 models ("assertion failed (actual=1)"), followed by registry-list, promote-force, confusion-matrix-format, and algo-logistic-regression, each failing 10 models. The promotion, registry listing, and ensemble-staging logic form the shared wall that no model clears.

Performance gap

The round table is a complete wall of zeros across all 12 models, so case pass is the only meaningful axis. Opus-4.8-xhigh leads at 97% and Qwen3.7-Max reaches 89% — both build essentially the entire orchestrator but fall at the promotion/ensemble edges driven by registry-promote and ensemble-stage. GPT-5.5 reaches 84%, then a cluster follows: Kimi-K2.7-Code (69%), DeepSeek-V4-Pro (62%), Kimi-K2.6 (61%), MiniMax-M2.7 (56%), and GLM-5.2 (55%).

The weak tier loses the pipeline core itself. GLM-5.1 (39%), Qwen3.6-Plus (40%), and DeepSeek-V4-Flash (32%) fail cross-validation and DAG execution well before the registry layer. MiniMax-M3 (0%) fails entirely with no test cases passing at all. The gap is layered: the top three are within a handful of requirements of a clean run that the binary round reward erases entirely, while the bottom four never get the core stages working.

Is the task fair? (top-model check)

Claude Opus-4.8 passed 47 of 48 cases in Round 1 but could not clear registry-promote — the check returned exit code 1 where valid-promotion acceptance was expected — and scored 0 rounds across all 7 due to the all-or-nothing reward structure. The oracle passes every round, confirming the promotion-validation logic is both correct and testable; Opus's miss reflects a genuine implementation gap in handling invalid-promotion rejection rather than any defect in the task or grading. Weaker models failing on this requirement are therefore a capability story, not a grading artifact.
